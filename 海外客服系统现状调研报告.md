## 2. 调研背景

当前客服系统已经演进为多模块、多服务的组合体系，不再是单一客服应用。系统覆盖多渠道接入、会话处理、机器人集成、规则引擎、人工客服、工单系统、知识库、历史检索等能力，但业务能力分散在多个模块中，缺少统一的产品视角描述。

本次整理目标是从“系统架构、业务流程、接口能力、数据模型、人工协同、知识库、非功能约束”几个维度，把当前系统状态整理成一份可用于沟通和后续规划的标准文档。

------

## 3. 调研结论摘要

### 3.1 总体判断

当前系统本质上是一个“多渠道接入 + 核心会话引擎 + 机器人/规则 + 工单 + 知识库 + 历史检索”的客服平台。

### 3.2 当前能力画像

| 维度         | 结论                                                         |
| :----------- | :----------------------------------------------------------- |
| 多渠道接入   | 已确认 已支持 Facebook、WhatsApp、Instagram、Line、Wechat、IM、TRTC、ChatApp 等多种渠道 |
| 实时消息链路 | 已确认 存在 Redis 消息隧道，并配合 Kafka、RabbitMQ 完成消息与事件流转 |
| 智能化能力   | 已确认 有机器人问答、推荐答案、AI 会话总结、AI 质检、AI 工单摘要 |
| 人工协同     | 已确认 有转人工、推荐答案、手动/自动发送模式、会话备注、小结、标签 |
| 售后承接     | 已确认 有工单创建、评论、分配、状态更新、标签、告警能力      |
| 电商业务能力 | 部分存在 订单信息可通过外部接口查询；物流、退款、退货、优惠券等能力未在本仓中发现闭环实现 |
| 知识体系     | 已确认 同时存在工单知识库与独立知识库模块，底层为数据库 + ES 检索 |
| 前端工作台   | 推断 主工作台是独立 Webapp，不在当前仓库主代码中             |

### 3.3 核心风险

- 业务能力分散，主数据不统一

- 会话 ID、用户 ID、渠道 ID 命名不完全统一

- 电商业务接口依赖外部系统，当前仓库难以形成完整闭环

- 非功能指标如 SLA、峰值并发、数据安全制度缺少集中说明

## 4.4 聊天消息链路

### 标准消息链路

1. 外部渠道消息进入 kefu-gateway 的各个 routeway-service

1. 网关将消息转换为统一结构并写入 Redis 消息隧道

1. 核心 kefu/services-message 消费消息并执行处理链

1. 处理链完成访客识别、会话创建、路由分流、消息持久化、机器人处理

1. 后续消息可能流向人工客服、规则引擎、工单系统、历史服务、知识服务

1. 出站消息由核心回推到网关，再由网关发回渠道侧

### 中间件形态

| 类型      | 使用情况                             |
| :-------- | :----------------------------------- |
| Redis     | 实时消息通道、状态缓存、推荐答案缓存 |
| Kafka     | 事件流、操作记录、审批、超时等主题   |
| RabbitMQ  | 通知、推送、部分异步链路             |
| WebSocket | TRTC 网关、工单消息推送              |
| SSE       | 未发现直接证据                       |

## 5. 核心业务流程

## 5.1 售前流程

### 当前实现

- 访客通过外部渠道发起咨询

- 系统优先进入机器人或规则处理

- 若命中知识答案则直接回复

- 若需要人工则触发转人工

- 若访客为 VIP，可走优先调度逻辑

### 自动化能力

- 机器人自动回复

- 菜单引导

- 规则匹配

- 推荐答案

- 知识检索

- VIP 优先调度

### 人工参与点

- 复杂咨询人工接入

- 人工判断是否继续转工单

## 5.2 售中流程

### 当前实现

- 已建立会话进入人工坐席处理

- 坐席可查看历史消息、推荐答案、会话标签、备注

- 协作模式可将推荐答案直接发给访客，或仅作为建议展示给坐席

### 自动化能力

- 推荐答案

- AI 会话总结

- 会话标签辅助

- 系统消息生成

### 人工参与点

- 人工正式回复

- 备注、小结、标签确认

- 是否发出协作答案的人工判断

## 5.3 售后流程

### 当前实现

- 人工或系统可创建工单

- 工单可被分配、评论、改状态、加标签、设置帮助主题

- 部分工单支持 AI 摘要

- 系统有告警与质检链路

### 自动化能力

- 工单系统 token 登录与远程转发

- 工单 AI 摘要

- 历史与评论检索

- 质检与 AI 质检

### 人工参与点

- 工单实际处理

- 工单分派

- 状态推进

- 投诉申诉判断

- 复杂售后决策

## 5.4 自动化与人工边界

| 场景         | 自动化程度 | 备注                         |
| :----------- | :--------- | :--------------------------- |

| 投诉升级     | 低         | 没有明确独立投诉域           |
| 退款退货     | 低         | 未在本仓发现实现             |

## 13. 结论与建议

## 13.1 结论

当前客服系统已经具备较完整的平台骨架，核心链路清晰，模块边界基本成立。系统强项在于“渠道接入、消息处理、会话流转、机器人协作、工单接入、知识搜索、历史沉淀”。系统短板主要集中在“外部业务系统依赖较重、主数据口径不统一、业务侧售后能力未在客服内核中沉淀完整”。
